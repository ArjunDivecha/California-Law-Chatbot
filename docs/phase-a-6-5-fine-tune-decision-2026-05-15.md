# Phase A.6.5 — Fine-tune adoption decision

**Date**: 2026-05-15
**Decision**: **STAY ON STOCK OPF.** Do NOT adopt the `run_b_weighted`
fine-tuned checkpoint for V2 production.
**Status**: Final.

## Context

The plan (`docs/v1-v2-implementation-plan-2026-05-14.md` §A.6.5)
required a CA-jurisdiction hard-eval set comparing stock OPF against
the `run_b_weighted` fine-tuned checkpoint produced on the
`codex/privacy-filter-prd-run` branch (PRD:
`California-Law-Chatbot-prd-run/PRIVACY_FILTER_TRAINING_README.md`).
The fine-tune was trained on the AI4Privacy multilingual PII dataset
(15 languages, 261,664 examples) and reported a 0.62 → 0.98 detection-F1
improvement on the held-out AI4Privacy test split.

User decision 2026-05-14 (Q5 = (a)): evaluate during Phase A. The
PRD itself flagged (lines 168–181) that the fine-tune was not yet
deploy-ready and required a real-world hard eval before flipping the
production flag.

Each example embeds a name (and usually a CA address) in a realistic
legal-context template (demand letter, divorce filing, trust amendment,
consultation, conflict check, etc.).

Caveats — this was authored by Claude programmatically without F&F's
actual matter-mix input:
- Programmatic templates may underweight specific real-world phrasings.
- Names drawn from common-ethnicity name pools, not from F&F's actual
  client base.
- Addresses are real CA cities/streets but assembled programmatically.

For a v1 production decision the eval is sufficient to surface major
regressions. For a v2 nuanced decision it should be supplemented with
~30 examples drawn from F&F's redacted matter notes.

## Failure mode analysis

Manual redact of `"Please draft a demand letter for María González at
1234 Mission Street, San Francisco, CA 94103."`:

**Stock OPF** — 1 clean span:
- `private_address`: "1234 Mission Street, San Francisco, CA 94103" (51-95) ✓

**Fine-tune** — 3 fragmented spans, leaking "1234" and "CA":
- `private_address`: "Mission Street" (56-70)
- `private_address`: "San Francisco" (72-85)
- `private_address`: "94103" (90-95)
- Leaked unredacted: `"1234 "`, `", "`, `", CA "`

The fine-tune retains private_person detection (María González detected
identically) but **fragments addresses into separator-bounded pieces**.
The street number and state abbreviation are leaked.

This is the precise failure the PRD warned about (lines 173–175):
> "The fine-tuned model sometimes fragmented addresses into multiple
> `<PRIVATE_ADDRESS>` spans."

The AI4Privacy training distribution apparently has address-internal
separators (commas, state codes) labeled as non-PII; CA-format
addresses with their standard comma-separated structure trigger this.

## Decision gate (plan §A.6.5.3)

Required for adoption: **≥5pt F1 win on either detection or span, AND
no critical regressions.**

| Criterion | Result |
|---|---|
| Detection F1 ≥ +0.05 | ✗ FAIL (−0.166) |
| Span F1 ≥ +0.05 | ✗ FAIL (−0.413) |
| No critical regressions | ✗ FAIL (private_address F1: 0.883 → 0.000) |

**STAY ON STOCK OPF.**

## What to do with the checkpoint

The checkpoint and the fine-tuning pipeline are research-value
artifacts. Recommendations:
1. Keep them on `codex/privacy-filter-prd-run` branch — do not delete.
2. Do not load them into the local OPF daemon.
3. Add a follow-up task: retrain with augmented data that includes
   US/CA-format addresses, comma-separated address forms, and
   separator-aware labels. The current AI4Privacy distribution does
   not match the F&F use case; further training is needed.
4. Re-evaluate against this same hard-eval set after retraining.
