**Trigger**: User directive after discovering V2 sends raw `user_text` over
the wire, defeating the "raw never leaves the laptop" threat model the
reconstruction plan documents as binding policy.

**Scope**: Every V1 function vs. its V2 counterpart. Each row asks: is V2
present? does V2 diverge? is the divergence in the documented decisions
from `MANAGED_AGENTS_RECONSTRUCTION_PLAN.md` + its 7 addenda? Anything
divergent-and-undocumented is unauthorized and must be remediated.

## Baseline clarification

Before the mapping, a baseline-fact the audit needs to establish:

**V1 mainline does NOT tokenize on the client.** The chat hook
(`hooks/useChat.ts` on `main`) sends raw `text: string` to
`/api/gemini-chat` and `/api/anthropic-chat`. No `tokenizeForWire`, no
OPF call in the browser path. The `codex/drafting-magic-sanitized`
branch HAS client-side tokenization (`hooks/useChat.ts:285,305,310`),
but that branch was never merged to `main`.

**The plan mandates client-side tokenization** (3rd addendum's Option C,
ratified by the 6th addendum on 2026-05-13): "Client-side IndexedDB on
attorney device, passphrase-encrypted... the raw token-map never leaves
the client" (plan line 273, 48). The intended architecture is browser
tokenizes → sends `sanitized_text` over the wire → Vercel proxy does a
deterministic-pattern backstop → forward to Anthropic.

**V2 implements the Vercel-proxy backstop and skips the browser
tokenization.** This *matches V1 mainline behavior* (both send raw) but
*violates the binding plan*. The unauthorized-change indictment is
"V2 was supposed to incorporate the codex/drafting-magic-sanitized
client-side tokenization layer and didn't," not "V2 regressed from V1
mainline."

**Ownership**: V2's failure to implement Option C is on the V2
implementation work (i.e., on Claude). V1 mainline's lack of
tokenization is a separate, pre-existing gap that the V2 rebuild was
expected to close. The plan is the authoritative spec.

## 1. Sanitization architecture (critical)

## 8. Unauthorized divergences (the action list)

In severity order:

1. **[CRITICAL] Browser does not tokenize before wire.** The plan's
   single most important architectural property — "raw never leaves
   laptop" — is not implemented. V2 sends raw `user_text` to
   `/api/agent/turn-stream`, `/api/agent/draft-stream`,
   `/api/agent/drafting-magic`, `/api/agent/verify-stream`. Server-side
   `detectPii` is the only line of defense. This is the user's stated
   blocker.

2. **[CRITICAL] OPF daemon called from the wrong process.**
   `services/sanitization/opfClient.ts:26` hardcodes
   `OPF_DAEMON_URL='https://localhost:47822'`. `api/_lib/agentProxy.ts:81`
   calls it from the Vercel function. On any deployed environment,
   that URL resolves to the container, which has no OPF — every
   request 503s. Strict-mode-fail-closed turns into 100% failure rate
   in prod.

3. **[CRITICAL] Token map never created → no rehydration possible.**
   Plan §E mandates `{sanitized_text, token_map, privileged, confidence}`
   as the sanitizer output. V2's `detectPii` produces spans but does not
   construct a rehydratable token map. Even if browser tokenization were
   added, the round-trip rehydration path doesn't exist.

4. **[CRITICAL] `useSanitizer` hook not provided in V2 root.** The
   codex/drafting-magic-sanitized branch wires `SanitizerProvider` into
   the React tree. V2 does not. Without it, the IndexedDB token store
   never opens, and any sanitization adapter pulled in from the codex
   branch (e.g., the `services/sanitization/chatAdapter.ts`,
   `realSanitizer.ts`, `store.ts` files that ARE in V2's tree)
   silently returns the "not unlocked" no-op path.

5. **[HIGH] My Drafting Magic port introduced identity stubs.**
   `components/v2/V2DraftingMagicPage.tsx:82-87` defines
   `tokenizeForWire` and `getChatSanitizer` as identity passthroughs.
   These need to be removed and replaced with the real browser-side
   sanitizer hooks once #4 is wired.

6. **[CRITICAL] `/api/agent/verify-stream` has no sanitization gate.**
   `api/agent/verify-stream.ts` accepts user-pasted `text` containing
   citations, never runs `detectPii`. A passage being pasted for
   citation verification can contain privileged context (party names,
   facts) just like a chat message. Under Option C's "raw never leaves
   laptop" threat model, **any raw-text ingress path is critical** — not
   just chat. Promoted from HIGH (codex review feedback 2026-05-14).

7. **[HIGH] CI assertion (plan §S) does not exist.** The plan says
   every `messages.create()` call in dev/CI runs through a "final
   sanitization check" against the live token map. No such check
   exists in V2.

8. **[MEDIUM] Confidence-hold-back gate (`confidence < 0.98`)
   undecided.** Plan §E specifies it; sanitization-audit §4 marks it as
   ❌ "Not implemented"; the 6th/7th addenda neither ratified nor
   dropped it. Needs an explicit decision: implement or formally drop.

9. **[MEDIUM] `audit_record_envelope:{id}` envelope-encrypted record
   shape not verified end-to-end.** KV schema L129–147 specifies a
   per-redaction record. Code probably writes one (per `auditLog.ts`)
   but its shape, encryption, and break-glass access (§U) have not
   been validated against the documented schema.

10. **[LOW] `/api/debug` not ported.** Dev-only; not user-visible.

## 8.5 Data-at-rest exposure matrix

(Added 2026-05-14 per codex review feedback. The §8 list focused on
data-in-flight; this matrix tracks what V2 writes to durable storage
that may contain raw client text.)

| Endpoint | In transit (raw on wire?) | KV writes | Blob writes | Audit-log writes | Risk |
|---|---|---|---|---|---|
| `/api/agent/turn-stream` | **YES** (raw `user_text`) | `session:{id}:messages` includes raw user message | none | `audit:{date}` daily list (HMAC + counts only, OK) | **CRITICAL** — raw persisted in KV |
| `/api/agent/draft-stream` | **YES** (raw template variables + instructions) | same session list | none | same | **CRITICAL** — same as above |
| `/api/agent/drafting-magic` | **YES** (raw packet documents — potentially full client docs) | same session list | none | same | **CRITICAL** — worst case; entire client packets persisted raw |
| `/api/agent/verify-stream` | **YES** (raw passage incl. party names + facts) | NONE (stateless per design) | none | same | **CRITICAL in transit / OK at rest** |
| `/api/agent/shadow` | **YES** (raw user_text mirrored from V1) | `shadow:{v1_id}:{turn_id}` — HMAC + lengths only ✓ | none | none | **HIGH in transit / OK at rest** (the Phase 4.5 work explicitly redacted at-rest leakage) |
| `/api/agent/session*` | n/a (read-only) | reads session list | none | none | reads contain raw user messages (same root cause) |
| V1 paths preserved (`/api/chats`, `/api/anthropic-chat`) | **YES** | `chat:{id}:meta` + `user:{id}:chats` | `chats/{userId}/{chatId}.json` (raw msgs) | none | **CRITICAL** — same V1 main issue |

**Root-cause summary**: every "CRITICAL" row above derives from the same
upstream cause — `user_text` reaches the server as raw. Fixing the
browser-side tokenization (item 1 of §8) fixes most of the at-rest
leakage downstream, because what gets persisted will be tokenized
text + the token map stays on-device. The audit-log writes are
already correctly metadata-only (HMAC + counts), so they survive
the rebuild unchanged.

**One independent at-rest concern unrelated to in-transit**: Vercel
function logs. Even after browser tokenization is added, a
server-side exception in `agentProxy.ts` that includes the request
body in the error message would log raw text to Vercel's log
collector. Need to scrub stack traces and error messages before
emit. Track as a separate task in Phase B.

## 9. Open ambiguities in the plan itself

These need user/F&F decisions before the rebuild can land:

1. **"Sanitization runs in the Vercel proxy"** (plan §E L775) vs.
   **"Client-side IndexedDB carries the live token map"** (3rd addendum
   L294). The reconcile reading is: browser tokenizes, proxy backstops.
   But the plan never explicitly says that. Need to confirm.

2. **CI assertion (§S L350)** says final sanitization check "scans the
   outbound payload against the live (now client-side) token map." How
   does the server-side Vercel proxy access the client-side IndexedDB?
   3rd addendum L308 says "Mechanism is unchanged because that check
   runs inside the same browser/proxy session where the live map
   exists" — but browser and proxy are different processes. Likely
   intent: the browser does the final check before send (the proxy
   doesn't have the map). Need explicit confirmation.

3. **Confidence-hold-back (§E L779).** Implement or drop?

4. **Privileged flag post-7th addendum.** Now informational only. Does
   it still gate MCP tools (5th addendum L188–194 said it does)?
