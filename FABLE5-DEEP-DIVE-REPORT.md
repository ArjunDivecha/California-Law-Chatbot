## 1. Executive summary

1. **V2 is a privacy-first legal-AI workbench whose core invariant — client identifiers never leave the attorney's laptop — is actually implemented and empirically gated**: browser-side GLiNER tokenization (`services/sanitization/detectionPipeline.ts`, `hooks/useSanitizer.tsx`), a pre-fetch wire guard (`services/sanitization/wireGuard.ts`), a fail-closed server regex backstop (`api/_lib/agentProxy.ts`), tool-output re-sanitization (`api/_lib/agentLoop.ts:sanitizeToolOutput`), and a 120-trap zero-wire-leak gate that passed 120/120 again on 2026-06-10 (`reports/traps-baseline-2026-06-10.json`).
2. **The engine moved to Claude Fable 5 yesterday/today (2026-06-12), as uncommitted work**: `DEFAULT_MODEL = process.env.V2_PRIMARY_MODEL ?? 'claude-fable-5'` in `api/_lib/agentLoop.ts`, `agents/california-legal/agent.json` (`max_tokens` 8192, `max_iterations` 12). Sonnet 4.6 stays for Quick Answer and the citation-verifier sub-agent. The brand-new `statute_verify` tool (leginfo / Cornell LII / eCFR) is also uncommitted.
3. **V1→V2 was a real architectural rewrite, not a reskin**: V1 main (frozen at `faf54d4`, 2026-05-16) was a Gemini 3.1 Pro Preview generator + Claude Sonnet 4.6 verifier, both via OpenRouter, plus a direct-Anthropic Speed mode — orchestrated by a 3,085-line `chatService.ts`, with **no sanitization on the wire at all** (raw text to OpenRouter/Anthropic/OpenAI/CourtListener/Serper, chat transcripts in *public-access* Vercel Blobs guarded only by URL secrecy — `utils/chatStore.ts:166`). V2 replaced it with one ~1,200-line agent loop on the Anthropic Messages API, seven in-process tools, tokenized-at-rest Upstash KV sessions, and the sanitization-first wedge.
4. **The governance trail is unusually strong**: a single plan document with ten dated addenda (`docs/MANAGED_AGENTS_RECONSTRUCTION_PLAN.md`) records every reversal (no-ZDR Team-plan posture, Option C device-side token map, web_search gate dropped, GLiNER over OPF, *Morgan v. V2X* contractual second net), each with partner sign-off where binding (`docs/partner-signoff-packet-2026-05-16.md`).
5. **Two pre-cutover defects matter most.** (a) **The inference endpoints are unauthenticated**: `/api/agent/turn-stream`, `draft-stream`, `drafting-magic`, `verify-stream`, `revise-section`, `shadow` accept `user_id` from the request body with no Clerk check and `Access-Control-Allow-Origin: *` — anyone who finds the URL can spend Fable tokens and write into any session (plan §A.1 explicitly requires JWT + CORS allowlist; only the *read* endpoints comply). (b) **A Fable refusal renders as nothing**: `stop_reason: "refusal"` produces an empty assistant bubble — the only trace is a gray `stop=refusal` footer (`components/v2/V2ChatPage.tsx:414-421`); §6 below specifies the single-engine fix.
6. **Fable 5's 1M context dissolves V2's biggest remaining product constraint** — today every surface is paste-sized (one document, top-K CEB chunks, 8–12 tool rounds). A whole-matter workspace (ingest the entire client file once, tokenized on-device, cached server-side) upgrades chat, drafting, Drafting Magic and Verify simultaneously and reuses the existing sanitization pipeline unchanged. This is the highest-leverage build (§5).
7. **The memory tool composes perfectly with Option C**: the model only ever sees tokens, so any memory it writes is token-space *by construction*; persisted matter memory (Upstash for standard matters, rehydrated only in the browser) gives cross-session matter continuity without a new privilege surface — and actually reduces exposure by ending re-pasting.
8. **Several Fable/API features are free wins V2 doesn't use yet**: prompt caching (V2 sends ~26K-token research turns with zero `cache_control`; at $10/MTok input this is the single biggest cost lever), structured outputs (`output_config.format` would replace the regex-parsed draft-proposal JSON in `components/v2/V2DraftPage.tsx` and the `VERDICT:` line-parsing in `api/_lib/verifierSubAgent.ts`), adaptive thinking + `effort` (V2 currently sends no `thinking` parameter at all), and the Batch API (50% off nightly citation sweeps).
9. **Vision is the one capability that is genuinely blocked by the zero-leak invariant**: pixels bypass every existing defense (the wire guard regex-scans strings; base64 images sail through). Scanned-exhibit support requires an on-device OCR+redaction stage in the GLiNER daemon first — proposed design in §4.C, sequenced last and gated on its own trap-suite extension.
10. **Cutover status**: Phase 4.5 shadow run is wired and firing (`api/agent/shadow.ts`, `reports/shadow-verify-2026-05-16.json`), partners signed 2026-05-16, prod V2 tested end-to-end 2026-05-18 — but `/` still lands on V1 (`App.tsx` routes), Phase 5a has not flipped, ~7,800 lines of V1 code await Phase 5b deletion, and the Fable-5 switch sits uncommitted on the worktree alongside untracked compliance docs.

## 2. Current-state report

### 2.1 Conceptualization & problem framing

**The problem.** Femme & Femme LLP (two-partner California family/probate/civil practice — Rachel and Lyla per `docs/morgan-memo-to-rachel-lyla-2026-06-02.md`) wants frontier-model legal research and drafting without the two failure modes that end careers in legal AI: **fabricated authority** filed with a court, and **client-confidential information leaking** to a third-party model provider in a way that breaches the duty of confidentiality (Cal. RPC 1.6) or risks privilege waiver. The pivotal trigger document is V1's `F&F_Meeting_Memo_2026-04-24.md`: *"Neither provider today offers contractual Zero Data Retention terms sufficient to satisfy California Rule of Professional Conduct 1.6 and ABA Formal Opinion 512… the tool is safe for general legal research, but not for client matters."* V2 exists to make the tool safe for client matters.

**Non-negotiable constraints, as documented:**

- **Privilege/confidentiality**: client identifiers must not reach Anthropic in raw form. After ZDR/enterprise paperwork was ruled out (2nd addendum, 2026-05-10: F&F stays on the Team plan), sanitization became the *primary* control. The 10th addendum (2026-06-02, *Morgan v. V2X*, 2026 WL 864223 (D. Colo.)) corrected the over-pessimistic "sanitization is the only line of defense" framing: Anthropic's Commercial Terms + DPA are a contractual second net (no-train, no-disclose, delete-on-request) that satisfies *Morgan* at $0 — sanitization remains the privilege/waiver minimizer.
- **Hallucination**: every case citation is verified against CourtListener and every statutory/regulatory citation against the official source (leginfo / Cornell LII / eCFR) — `api/_lib/tools/citationVerify.ts`, `api/_lib/tools/statuteVerify.ts`, `api/_lib/verifierSubAgent.ts`.
- **Defensibility**: 7-year metadata-only audit chain (HMAC of the sanitized prompt, never plaintext — `api/_shared/auditLog.ts`), per-redaction AES-256-GCM envelopes, evidentiary-discovery hardening (§W of the plan: system prompts treated as legal artifacts), UPL controls (§X), and a planned §Y per-session attestation generator.
- **FAIL IS FAIL**: no silent fallbacks. The GLiNER daemon down ⇒ no chat (`opfClient.ts` removed the stock-OPF fallback because "a silent fallback to a lower-quality detector violates" the partner-ratified posture); raw PII at the server ⇒ HTTP 503, never a quiet send (`agentProxy.ts`).
- **Single engine**: "No fallback inference provider… Bedrock and OpenRouter are explicitly out of scope" (1st addendum). The same policy now applies model-level: Fable 5 is the engine; refusals must surface, never silently reroute.

**Design philosophy reconstructed from the docs.** Three ideas recur across the plan and its ten addenda:

1. **The wedge is sanitization-first, everything else is replaceable.** The 4th addendum's "portability principle" deliberately shaped V2's agent definitions (`agents/california-legal/agent.json`, Skill markdown files) to mirror Anthropic's plugin format so the runtime could later be swapped without rewriting the product. The tools and prompts changed repeatedly; the wedge never did.
2. **Detectors inform, attorneys decide.** The 7th addendum dropped the web_search privilege gate ("the heuristic-based decision to block web_search was overreach by code"), the 8th dropped the confidence-0.98 hold-back. Detection results became chips (`components/v2/V2SanitizationChip.tsx`, the amber "Privileged content detected" badge in `V2ChatPage.tsx`) plus audit-record fields — never behavior gates. The one deliberate exception is court-order-driven (§Z protected-discovery mode, designed but dormant).
3. **Empirical gates over assertions.** The §0.c hard gate (zero wire-leaks across two consecutive 120-trap runs) blocked the migration until GLiNER cleared it (9th addendum); the Phase 3 verifier shipped only after F1 = 0.919 / fake-caught = 1.000 on a 30-citation eval (`reports/phase3-eval-sub-agent-2026-05-13.json`); the subscription-billing idea died by smoke test, documented as a dead-end so it never gets re-litigated (`README.md` §"Architectural dead-ends").
4. **Honest uncertainty beats confident output** — a direct reaction to V1, whose verifier prompt said *"NEVER set status to 'refusal'"* and whose code silently restored the original answer when the verifier balked (`verifierService.ts:340-347` on V1 main). V2's verifier instead has a first-class `ambiguous` verdict ("we don't know" ≠ "fake", `verifierSubAgent.ts`), and the same lesson is why proper *refusal surfacing* (§4.G) — not suppression, not fallback — is the policy-correct behavior for the Fable engine itself.

A fourth, painful lesson is recorded in `docs/v1-v2-audit-2026-05-14.md`: the first V2 cut **sent raw `user_text` over the wire**, violating the ratified Option C plan. The 2026-05-14 audit classified every divergence as documented-vs-unauthorized, and Phases A–C rebuilt the browser-side pipeline in four days (commits `27d8b27` → `bc3677f`), ending at a 25-item compliance scorecard with 23 ✅ (D15, the envelope writer, landed afterward in `222ac72` — `writeRedactionEnvelope` in `api/_shared/auditLog.ts`).

Key structural facts:

- **One file talks to Anthropic.** `api/_lib/agentLoop.ts` (runTurn / runTurnStream) is the single `messages.create`/`stream` call site; `agentProxy.ts` wraps it with the backstop and audit writes; five thin routes under `api/agent/` consume it. - **Model roles**: Research/Draft/Magic = Fable 5 (`V2_PRIMARY_MODEL` env-overridable); Quick Answer = Sonnet 4.6 with zero tools; verifier sub-agent = Sonnet 4.6 (`V2_VERIFIER_MODEL`), fresh conversation per citation, ~18s/cite. No `thinking` parameter is sent anywhere (legal on Fable — but adaptive thinking is being left on the table; see §4.H).
- **System prompts are data**: composed per-turn by `api/_lib/skills.ts` from `agents/california-legal/skills/*.md` (core + at most one intent skill via keyword routing; drafting skills selected by `template_id`). Format intentionally mirrors `anthropics/claude-for-legal`.
- **Sessions**: Anthropic-shaped message arrays in Upstash KV, append-only with `turn_id`/`sequence`; tool-result idempotency cache (24h) stores the *sanitized* form so cache hits can't bypass the wrapper (`dispatchWithCache`); single-flight lock helpers exist (`acquireLock`) but are not called by any route yet.
- **Deployment**: Vercel (`vercel.json`: 300s maxDuration on agent routes, `includeFiles: agents/**`, SPA rewrite, hardening headers added 2026-05-18). V1 and V2 coexist: `/` → V1 `NewChatRedirect` → `/c/:chatId`; V2 lives at `/v2*` (`App.tsx`). Cutover (Phase 5a flag-flip) has not happened.
- **The daemon** (`tools/gliner-daemon/gliner_daemon.py`): `urchade/gliner_multi_pii-v1`, threshold 0.7 (`GLINER_THRESHOLD`), ~150-entry stoplist, title/role prefix-trim, lazy load (~7s cold) with 10-min idle unload, loopback-only bind enforced, OPF-compatible label vocabulary so `opfClient.ts` needed no protocol change. 45ms median steady-state vs stock OPF's 1.6s (35× — `docs/phase-c-decision-2026-05-15.md`).

### 2.4 Codebase health

**Strengths.** The privilege-critical path is small, single-purpose, and heavily commented with *why*-grade comments referencing the governing addendum. The test/evidence culture is real: 120-trap manifest with per-trap `must_redact`/`must_not_redact`/`expected_privileged` schema (`tests/traps/manifest-v1.json`), a wire-level runner (`tests/traps/runTrapsWire.mjs`), ~40 probe/e2e scripts under `scripts/`, and 30+ JSON evidence reports under `reports/` — including the re-validated 2026-06-10 120/120 run. The 2,163-line unit suite (`tests/sanitization.test.mjs`) plus `yarn test:traps` are reproducible gates. Plan-vs-code drift, when it happened, was caught by an honest audit and remediated with exit criteria.

**Defects and risks, ranked. Items 1–2 are pre-cutover blockers; nothing below threatens the core invariant on the current single-attorney deployment, but several weaken the defensibility story the product sells.**

1. **[CRITICAL — security] Inference/write endpoints have no authentication.** `turn-stream`, `draft-stream`, `drafting-magic`, `revise-section`, `verify-stream`, `shadow` perform no Clerk verification and trust a body-supplied `user_id`; CORS is `*` both per-route and globally (`vercel.json`). Consequences: unmetered Fable-5 spend for anyone who discovers the prod URL; cross-user session pollution (POST a turn into any guessed/known `session_id` — `runTurn` appends without ownership check, and `registerSessionForUser` will happily index it); forged `user_id` in 7-year audit records. This posture is *inherited* — in V1 only `/api/chats` ever authenticated, and every AI/search endpoint was open with CORS `*` — which is precisely why plan §A.1 was written ("That is unacceptable for the new agent surface"). The read endpoints (`session.ts`, `sessions.ts`) already comply; the same ~20-line `getUserId` guard (centralized in `utils/auth.ts`) needs to be applied to the six write routes, plus session-ownership verification on `session_id`, plus the CORS allowlist. It is the largest documented-but-unimplemented item in the codebase.
2. **[HIGH — UX/policy] No refusal handling anywhere** (the single-engine policy's missing half). In `agentLoop.ts`, a `stop_reason: "refusal"` with empty content exits the loop with `final_text = ''`; the UI folds a message only `if (state.done && state.tokens)` (`V2ChatPage.tsx:263`), so the attorney sees a blank — just `stop=refusal` in a gray footer. Draft/Magic flows would emit an empty proposal list or a `quality_warning` at best; the verifier treats an unparseable output as `ambiguous` (good) but never inspects refusal specifically. Fable 5 returns structured `stop_details` (`category`, `explanation`) precisely for this. Fix specified in §6 — surface, explain, offer revise; never model-switch.
3. **[MEDIUM — sanitization scope] The verifier sub-agent bypasses the tool-output sanitizer.** `verifierSubAgent.ts:dispatchVerifierTool` returns raw JSON into `tool_result` blocks, while the main loop routes everything through `sanitizeToolOutput()` (2nd-addendum mandate: "every tool_result block runs through the same sanitizer"). Exposure is low (public-record sources, tokenized inputs) but it is a documented-control gap on the same class of risk the W5 traps test — one helper call fixes it (with the same caption-safe carve-out from Task #71).
4. **[MEDIUM — docs-vs-code trust model] The "passphrase" never shipped.** Option C and the 3rd addendum describe a *passphrase-encrypted* IndexedDB store with attorney-burden language; the implementation is a random device key in `localStorage` (`useSanitizer.tsx`, "intentionally no attorney-facing passphrase"). The §Q memo describes actual behavior honestly (device loss = disk-encryption problem; clear-site-data = unrecoverable), so partners signed the real design — but the plan text and audit-scorecard language ("passphrase-encrypted per attorney device") should be corrected, because §W assumes adversarial counsel will read these documents against the code.
5. **[MEDIUM — audit chain weaker than the plan sells]** (a) No SHA-256 hash chaining or signed daily digest (§G specifies both; `auditLog.ts` LPUSHes independent records). (b) Daily audit list TTL is 90 days vs §I's 7-year retention for audit logs (only the envelopes get 7 years). (c) The audit writer deliberately *fails open*. (d) The §Y attestation generator and `verify-attestation` CLI (Phase 6) don't exist yet, while partner-facing docs lean on the attestation chain. (e) In the redaction envelope, `input_sha256` and `sanitized_sha256` are both HMACs of the *same server-received (already-tokenized) text* (`agentProxy.ts:128-129`) — the browser never supplies a raw-input hash, so the Option C deposition claim "hash binding to the original input" is not yet true. Fix: compute the raw-input HMAC browser-side (key-less SHA-256 or a device-held key) and send only the hash.
6. **[MEDIUM — repo hygiene with evidentiary stakes]** The Fable-5 switch, `statute_verify`, verify-stream statute integration, plan 10th-addendum edits, installer changes — all uncommitted on the worktree (last commit `9063d37`, 2026-05-23); `docs/partner-signoff-packet-2026-05-16.md` and the Morgan memo are *untracked*. For a project whose §W posture is "every system-prompt change reviewed, full version history retained 7 years," three weeks of un-versioned changes to the privilege-critical path (with Dropbox as the only backstop) is a real gap. Commit, and wire `yarn test:traps` as the pre-merge gate the plan §S describes (no CI config exists in-repo).
7. **[LOW] Minor correctness/staleness:** `ConfidentialityAttestation.handleDismiss` calls `acknowledge()` — "Not now" permanently attests (contradicts its own comment). `RunTurnOptions.model` doc-comment says "defaults to claude-sonnet-4-6" (stale). Root `README.md`/`CLAUDE.md` still describe the V1 OpenRouter stack below the status header. `server backstop` scans `user_text` but not the client-suppliable `system_prompt` field (browser wire-guard covers it; server defense-in-depth doesn't). Client-minted `session_id` is interpolated into Redis keys unvalidated (`v2_…` convention not enforced — harmless once auth lands, worth a `^[\w-]+$` check). ~7,800 lines of V1 code plus root-level `test-*.js` clutter await Phase 5b. `cancel()` exists in `useV2AgentStream` but no Stop button uses it; the server runs the turn to completion regardless (by §D design, but the UI should say so).
8. **[Residual risks accepted & documented]** Compound-identifier prompts ("Cantonese-speaking widower in Sunset District…") are *flagged*, not tokenized — attorney judgment is the final filter (§Q memo "Limits of the design" #4). A public-record name identical to a client's can transit inside case captions (the deliberate Task #71 caption carve-out in `agentLoop.ts:CAPTION_SAFE_FIELDS`). Anthropic could change Team-plan retention (quarterly review suggested). These are honest, partner-ratified trade-offs rather than defects.

## 4. The Fable 5 opportunity (Part 2 — the main event)

Context for every proposal: Fable 5 = 1M-token context, 128K max output, $10/$50 per MTok, adaptive-thinking-only surface (no sampling params; explicit `thinking:{type:"disabled"}` 400s — omit instead), structured `stop_details` on refusals, `effort` low→max, Task Budgets (beta), server-side code execution + programmatic tool calling, client-implemented memory tool (`memory_20250818`), compaction (beta `compact-2026-01-12`), structured outputs, Batch API at 50%. ### A. 1M context — the Whole-Matter Workspace (top pick; build sketch in §6)

**Today's ceiling**: every surface is single-paste-sized. Drafting Magic packets are individually tokenized but practically bounded by attention and cost at ~tens of KB; chat gets top-5-per-namespace CEB chunks; the model never sees "the matter," only the excerpt in front of it.

**The unlock**: ingest the *entire client file* — the trust instrument set, the full correspondence history, both prior depositions, the draft MSA, opposing counsel's letters — once, on-device-tokenized per document, and hold it as a persistent, prompt-cached matter context that every surface (chat turn, draft proposal, Magic run, verify pass) reasons over in full. ~3,000 pages ≈ 750K–900K tokens fits in one window. New workflows this enables that chunking cannot: cross-document inconsistency sweeps ("every place the trust and the pour-over will disagree"), deposition-wide contradiction finding with pin-cites, "what did we already tell this client about X" grounded answers, redlines that respect the whole agreement rather than the visible section.

**Zero-leak**: unchanged pipeline, larger payload. Each document runs through the existing `extractTextFromFile` → chunked `detectSpans` → `tokenizeWithSpans` path; token-map stability guarantees `CLIENT_001` is the same entity in every document; `assertNoRawPii` scans the assembled bundle. One real engineering risk: GLiNER's 45ms benchmark is for short texts — a 500KB document needs a chunked detection pass with offset stitching and a progress UI (daemon already accepts 10MB bodies; inference must be windowed). The trap suite needs long-document traps (PII at deep offsets, cross-chunk name splits) before the gate re-runs.

**Cost / latency** (why prompt caching is a prerequisite, not an optimization): a 300K-token matter at $10/MTok is $3.00 of input *per turn* uncached; with a 1h-TTL cache breakpoint after the matter block it's a one-time ~$6 write then ~$0.30/turn read. Without caching this feature is uneconomic; with it, it's cheap. **Effort: M** (browser store + ingestion UX + prompt assembly + caching + trap extension). **Risk: medium** (detection latency at size; KV 50KB-per-session assumptions need a matter-blob store — Vercel Blob for tokenized bundles fits).

### B. Long-horizon agentic work — the Verified Drafting Run

**Today**: drafting is one streamed pass, ≤12 tool rounds, inside one ≤300s Vercel invocation; citation verification is a separate manual step the attorney may skip.

**The unlock**: "Produce a motion to compel with verified authorities" as a *job*, not a turn: (1) research & outline → **checkpoint: attorney approves outline + authority list** (reuses the Draft page's propose→approve interaction); (2) full draft with §-structure; (3) self-verification — every citation through `citation_verify`/`statute_verify`, every flagged-fake authority replaced or removed; (4) adverse-authority sweep (§E); (5) final draft + attached verification report → **checkpoint: attorney review-attestation** (the §Y companion the plan already envisions). Fable-specific levers: `max_iterations` to ~40 for the job runner, `output_config.effort: "high"|"xhigh"`, Task Budgets so the model self-paces against a token cap the firm sets, and the up-front full-spec pattern Fable rewards.

**Plumbing reality**: this exceeds the 300/800s function ceiling — exactly the §C "async/queue split" the plan anticipated. Implementation: a `matter_job` record in KV + `waitUntil`-style background continuation or QStash-triggered worker, progress events appended to the session stream, UI resumes via existing session hydration. Each checkpoint is a hard stop persisted in KV — FAIL IS FAIL applies (a failed verification halts the job with the report, never ships a draft silently).

**Zero-leak**: unchanged — same wire, same tokenized inputs; long jobs mean *more* tool outputs through the same sanitizer. **Effort: L. Risk: medium-high** (queue infra + spend control; mitigated by Task Budgets and per-job caps). Sequencing: after A — a motion grounded in the whole matter file is the version worth shipping.

### C. Vision — scanned exhibits, filed PDFs, handwriting (blocked until the daemon grows eyes)

**The honest blocker first**: every existing defense is text-shaped. `assertNoRawPii` regex-scans strings; the backstop scans `user_text`; a base64 image block would sail past all of it carrying a client's name in pixels. Under "any new input path MUST tokenize before the wire," raw scans **cannot be sent**. Flag: *vision input is a blocker until an on-device redaction stage exists.*

**The design that preserves the invariant**: extend the GLiNER daemon (`gliner_daemon.py`) with `/v1/ocr_redact`: on-device OCR with word-level bounding boxes (macOS Vision framework via pyobjc, or tesseract bundled in the .pkg; PyMuPDF rasterizes PDFs) → OCR text through the *existing* GLiNER+regex+allowlist pipeline → two outputs: (a) tokenized extracted text (today's path), and (b) a **redacted image** — black boxes burned over every span box whose category is HIGH_RISK, plus an attestation `{page, boxes, categories, ocr_confidence}`. The browser may then send tokenized text alone (default) or redacted-image + tokenized-text (when layout matters: stamped filings, checkboxes, signature presence, handwriting margins). Wire-guard extension: *reject any image content block lacking a daemon redaction attestation*; new trap category (synthetic scans with planted PII, low-contrast text, rotated pages) gated to zero leaks before enable. Handwriting fails closed: OCR confidence below threshold ⇒ region is redacted entirely unless the attorney explicitly marks it public (consistent with the 7th-addendum "attorney decides" posture — but default-deny because undetected pixels are unrecoverable once sent).

**What it unlocks for this practice**: probate and family files are *scan-heavy* — filed orders with stamps, exhibit packets, handwritten codicils, intake forms. Fable's high-res vision (2576px long edge, pixel-accurate) reads what text extraction loses. **Effort: L** (the redaction stage is the product). **Risk: high** — sequenced last, gated on its own zero-leak evidence. Until then, the existing `fileTextExtraction.ts` text-only path remains the rule, and the UI should say *why* a scanned PDF with no text layer is rejected.

### D. Memory tool — matter-level memory that is sanitized by construction

**The reconciliation the prompt asks for is surprisingly clean.** The memory tool is client-implemented: the API emits `memory` tool_use commands; *V2's own loop* executes them against a backend V2 controls (`api/_lib/agentLoop.ts` dispatcher + a new `memoryBackend.ts`). Two properties make it compatible with the device-local trust model: (1) **the model can only write what it has seen, and it only ever sees tokens** — so model-authored memory is token-space by construction ("CLIENT_001 prefers mediation; opposing counsel is CLIENT_007's former partner"); (2) the token map's stability (`store.assignToken` returns the same `CLIENT_001` forever) means token-space memory stays *meaningful* across sessions on the attorney's device, where it rehydrates for display.

**Design**: per-matter directories (`/memories/matter_<id>/strategy.md`, `facts.md`, `preferences.md`) stored in Upstash KV for `standard` matters (same retention class as the tokenized `session:{id}:messages` already there; firm-store for `protected_discovery` per §Z.4). Belt-and-suspenders: run `analyze()` + compound-risk on every memory **write**, record a redaction-envelope-style audit entry, and destructively redact any HIGH_RISK span that somehow appears raw (e.g., the model copying a compound-risk plain-text fact). A Memory panel in the UI shows rehydrated contents with per-entry delete (attorney-reviewable, deletable — matching Anthropic's deletion-on-request posture). Memory-version history can reuse the audit chain.

**Why it matters beyond convenience**: today the attorney re-pastes matter context every session — each re-paste is a fresh detection pass and a fresh chance for a miss. Persistent memory *reduces* cumulative exposure while making turn 1 of session 30 as informed as turn 50 of session 1. Anthropic's launch data (memory helped Fable ~3× more than Opus 4.8) suggests Fable actually exploits it. **Effort: M. Risk: medium** (memory poisoning/staleness — mitigated by the review panel and per-matter scoping). Pairs naturally with A.
